= Artifact definition example

This defines an artifact that creates a Gitea runner token, demonstrating `files`, `prompts`, and `generator` options.

[source,nix]
----
{ config, pkgs, ... }:
{
  artifacts.store.gitea-runner = {
    files.auth_token = { # <1>
      path = "/var/lib/gitea-runner/auth_token"; <2>
      mode = "0400"; <3>
      owner = "gitea-runner"; <4>
      group = "gitea-runner"; <5>
    };
    
    prompts.auth_token.description = "Enter auth token"; <6>
    
    generator = pkgs.writeShellApplication { <7>
      name = "generate-gitea-auth-token";
      text = ''
        TOKEN="$(cat $prompts/auth_token)"
        echo "TOKEN=$TOKEN" > "$out/auth_token"
      '';
    };
  };
}
----
<1> the file handle `auth_token` - this name must match the filename created by the generator in `$out/`
<2> the xref:options-nixos.adoc#_artifacts_store_name_files_name_path[path] where `auth_token` should be stored on the target system
<3> the xref:options-nixos.adoc#_artifacts_store_name_files_name_mode[mode] of `auth_token`
<4> the xref:options-nixos.adoc#_artifacts_store_name_files_name_owner[owner] that owns the file
<5> the xref:options-nixos.adoc#_artifacts_store_name_files_name_group[group] that owns the file
<6> creates a xref:options-nixos.adoc#_artifacts_store_name_prompts_name_description[prompt] before the `generator` script is executed
<7> the xref:options-nixos.adoc#_artifacts_store_name_generator[generator] script definition

== Using artifact paths

Reference artifact paths in NixOS configuration using `config.artifacts.store.<name>.files.<file>.path`:

[source,nix]
----
{ config, pkgs, ... }:
{
  services.gitea-actions-runner = {
    instances."my-runner" = {
      enable = true;
      tokenFile = config.artifacts.store.gitea-runner.files.auth_token.path; <1>
    };
  };
}
----
<1> References the xref:options-nixos.adoc#_artifacts_store_name_files_name_path[path] defined in the artifact, creating a dependency.


== Generator contract

The generator must:

- Read optional user inputs from files in $prompts/<name> for each prompts entry.
- Write one file per files.<name> handle into $out/<name>.

The backend will then handle serialization of $out.

See also xref:backend-concepts.adoc[Backend Concepts] for how serialization/deserialization is provided.

== Shared artifact example

Shared artifacts are generated once and distributed to multiple machines. Set `shared = true` on each target that should receive the artifact.

[source,nix]
----
# flake.nix
{
  outputs = { self, nixpkgs, nixos-artifacts, ... }: {
    nixosConfigurations.server-one = nixpkgs.lib.nixosSystem {
      # ...
      modules = [
        nixos-artifacts.nixosModules.default
        ({ pkgs, ... }: {
          artifacts.store.oidc-secret = {
            shared = true;  # <1>
            files.client-secret.path = "/run/secrets/oidc-client-secret";
            generator = pkgs.writeShellApplication {
              name = "gen-oidc";
              runtimeInputs = [ pkgs.openssl ];
              text = ''
                openssl rand -hex 32 > "$out/client-secret"
              '';
            };
          };
        })
      ];
    };

    nixosConfigurations.server-two = nixpkgs.lib.nixosSystem {
      # ...
      modules = [
        nixos-artifacts.nixosModules.default
        ({ pkgs, ... }: {
          artifacts.store.oidc-secret = {
            shared = true;  # <1>
            files.client-secret.path = "/run/secrets/oidc-client-secret";
            generator = pkgs.writeShellApplication {
              name = "gen-oidc";
              runtimeInputs = [ pkgs.openssl ];
              text = ''
                openssl rand -hex 32 > "$out/client-secret"
              '';
            };
          };
        })
      ];
    };
  };
}
----
<1> Both machines define xref:options-nixos.adoc#_artifacts_store_name_shared[shared] as `true` for the same artifact name `oidc-secret`.

=== How shared artifacts work

When you run the TUI:

1. The CLI aggregates all definitions of `oidc-secret` with `shared = true`.
2. The TUI displays a single entry for `oidc-secret` showing both targets (server-one, server-two).
3. When generated, the secret is created once and serialized for all targets using the backend's shared serialize script.

=== Generator selection

If different targets define different generator scripts for the same shared artifact, the TUI displays a selection dialog allowing you to choose which generator to use. Each option shows:

- The generator path (from the Nix store)
- The list of targets using that generator

This can happen when targets have slightly different generator configurations, for example due to different package versions or generation options.
