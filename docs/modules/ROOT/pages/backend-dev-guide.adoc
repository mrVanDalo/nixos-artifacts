= Backend Developer Guide
:description: Complete reference for implementing custom serialization backends for nixos-artifacts
:toc: macro

WARNING: The backend interface is experimental and may change in future versions.

== Introduction

This guide provides comprehensive documentation for implementing custom backends that integrate with nixos-artifacts. Backends are responsible for the serialization (storage) and deserialization (retrieval) of artifact files generated by your NixOS or Home Manager configurations.

=== What is a Backend?

In nixos-artifacts, a **backend** is a set of shell scripts that implement the storage contract for artifact files. When you run `artifacts generate`, the CLI orchestrates a workflow that:

1. Checks if artifacts already exist (to avoid accidental overwrites)
2. Runs generator scripts to create files
3. Calls your backend's `serialize` script to store the files
4. Later, during system activation, calls your `deserialize` script to restore them

=== When to Write a Custom Backend

You should consider writing a custom backend when:

* You want to integrate with a secret management system not yet supported
* Your organization has specific compliance or storage requirements
* You need custom encryption or access control mechanisms
* You want to store artifacts in a proprietary or internal system

=== Backend vs Generator

It's important to distinguish between these two concepts:

**Generators**::
  Create files from user prompts (e.g., generating SSH keys, creating SSL certificates)

**Backends**::
  Store and retrieve those files (e.g., agenix encryption, sops-nix, agebox)

Generators produce content; backends persist it.

=== Backend Types

Backends must handle two types of targets:

**NixOS Machines**::
  Full system configurations where artifacts may need specific ownership (owner/group) and permissions.

**Home Manager Users**::
  User-level configurations where artifacts are owned by the user and don't require system-level permissions.

**Shared Artifacts**::
  Artifacts that are identical across multiple targets and should be stored once, not per-target.

== Backend Interface Contract

Each backend implements one or more scripts that the CLI calls at specific points in the artifact lifecycle. All scripts run in a bubblewrap container with restricted filesystem access for security.

=== check_serialization Script

**Purpose:**
Determine whether an artifact already exists in your backend's storage. This prevents accidental overwrites of existing secrets.

**When Called:**
Before the generator script runs, during the `artifacts generate` workflow.

**Environment Variables:**

|===
| Variable | Description | Example Value

| `$inputs` | Directory containing JSON files with file metadata | `/tmp/artifacts-inputs-xxx`
| `$config` | Path to JSON file with backend configuration | `/tmp/config-xxx.json`
| `$machine` | Target identifier (machine name or `user@host`) | `server-one` or `alice@workstation`
| `$artifact` | Name of the artifact being processed | `ssh-host-key`
|===

**Input Details:**

The `$inputs` directory contains one JSON file per file defined in the artifact's `files` attribute. Each JSON file has this structure:

[source,json]
----
{
  "path": "/etc/ssh/ssh_host_ed25519_key",
  "owner": "root",
  "group": "root"
}
----

For Home Manager artifacts, `owner` and `group` are omitted (only `path` is present).

**Exit Codes:**

|===
| Exit Code | Meaning | Action Taken by CLI

| 0 | Artifact exists | Skip generation for this artifact
| Non-zero | Artifact needs generation | Continue to generator phase
|===

**Output Convention:**
While not required by the CLI, it's conventional to print `EXISTS` to stdout when the artifact is found. This aids in debugging.

**Example Implementation:**

[source,bash]
----
#!/usr/bin/env bash
set -euo pipefail

# Check if artifact exists in storage
STORAGE_PATH=$(jq -r '.storage_path // "/var/lib/mybackend"' "$config")

if [ -f "$STORAGE_PATH/$artifact.json" ]; then
    echo "EXISTS"
    exit 0
fi

exit 1
----

=== serialize Script

**Purpose:**
Store the files generated by the generator script in your backend's storage.

**When Called:**
After the generator script successfully completes, for each target that needs the artifact.

**Environment Variables:**

|===
| Variable | Description | Example Value

| `$out` | Directory containing all generated files | `/tmp/artifacts-out-xxx`
| `$config` | Path to JSON file with backend configuration | `/tmp/config-xxx.json`
| `$machine` | Target identifier | `server-one` or `alice@workstation`
| `$artifact` | Name of the artifact | `api-key`
|===

**Input Details:**

The `$out` directory contains the files created by the generator script. Filenames match the keys in the `files` attribute of your artifact definition.

**Expected Behavior:**

* Store all files from `$out` in your backend
* Use the backend configuration from `$config` for storage parameters
* Return exit code 0 on success, non-zero on failure
* The CLI aborts if this script fails

**Example Implementation:**

[source,bash]
----
#!/usr/bin/env bash
set -euo pipefail

STORAGE_PATH=$(jq -r '.storage_path // "/var/lib/mybackend"' "$config")
mkdir -p "$STORAGE_PATH"

# Store each generated file
for file in "$out"/*; do
    if [ -f "$file" ]; then
        filename=$(basename "$file")
        # Example: copy to storage (replace with your storage logic)
        cp "$file" "$STORAGE_PATH/$artifact-$filename"
    fi
done
----

=== deserialize Script

**Purpose:**
Restore stored artifact files for use during system activation.

**When Called:**
During NixOS system activation or Home Manager activation.

**Environment Variables:**

|===
| Variable | Description | Example Value

| `$inputs` | Directory with expected file metadata (same format as check_serialization) | `/tmp/artifacts-inputs-xxx`
| `$config` | Path to JSON file with backend configuration | `/tmp/config-xxx.json`
| `$machine` | Target identifier | `server-one` or `alice@workstation`
| `$artifact` | Name of the artifact | `database-password`
|===

**Output Requirements:**

Your script must create files in the `$out` directory (which doesn't exist yet - you must create it). The CLI will then copy these files to their final destinations with correct ownership and permissions.

**Expected Behavior:**

* Create the `$out` directory
* Restore all files expected by the artifact
* Place restored files in `$out` (filenames must match artifact `files` keys)
* Return exit code 0 on success

**Example Implementation:**

[source,bash]
----
#!/usr/bin/env bash
set -euo pipefail

STORAGE_PATH=$(jq -r '.storage_path // "/var/lib/mybackend"' "$config")

# Create output directory
mkdir -p "$out"

# Restore each expected file
for metadata_file in "$inputs"/*; do
    if [ -f "$metadata_file" ]; then
        filename=$(basename "$metadata_file")
        source_file="$STORAGE_PATH/$artifact-$filename"

        if [ -f "$source_file" ]; then
            cp "$source_file" "$out/$filename"
        else
            echo "Error: Missing stored file for $filename" >&2
            exit 1
        fi
    fi
done
----

=== shared_serialize Script (Optional)

**Purpose:**
Store an artifact for all targets that share it, called once instead of per-target.

**When Called:**
After successful generator execution for artifacts with `shared = true`.

**Environment Variables:**

|===
| Variable | Description | Example Value

| `$out` | Directory containing generated files | `/tmp/artifacts-out-xxx`
| `$artifact` | Name of the artifact | `wireguard-key`
| `$machines` | Path to JSON file mapping machine names to configs | `/tmp/machines-xxx.json`
| `$users` | Path to JSON file mapping user@host to configs | `/tmp/users-xxx.json`
|===

**Note:** Unlike other scripts, `$config` is **not** directly available. Read per-target configuration from the `$machines` and `$users` JSON files.

**Input Details:**

The `$machines` JSON file maps machine names to their backend configurations:

[source,json]
----
{
  "server-one": {
    "storage_path": "/var/lib/mybackend",
    "encryption_key": "abc123"
  },
  "server-two": {
    "storage_path": "/var/lib/mybackend",
    "encryption_key": "def456"
  }
}
----

The `$users` JSON file has the same structure but for Home Manager targets:

[source,json]
----
{
  "alice@workstation": {
    "storage_path": "~/.local/share/mybackend"
  },
  "bob@laptop": {
    "storage_path": "~/.local/share/mybackend"
  }
}
----

**Expected Behavior:**

* Store the artifact for all machines listed in `$machines`
* Store the artifact for all users listed in `$users`
* Handle the case where one or both files may be empty (no targets of that type)
* Return exit code 0 on success

**Example Implementation:**

[source,bash]
----
#!/usr/bin/env bash
set -euo pipefail

# Store for each machine
if [ -f "$machines" ] && [ "$(jq 'length' "$machines")" -gt 0 ]; then
    for machine in $(jq -r 'keys[]' "$machines"); do
        machine_config=$(jq -r ".\"$machine\"" "$machines")
        storage_path=$(echo "$machine_config" | jq -r '.storage_path')

        # Store files for this machine with its specific config
        for file in "$out"/*; do
            if [ -f "$file" ]; then
                store_file "$file" "$machine" "$machine_config"
            fi
        done
    done
fi

# Store for each user
if [ -f "$users" ] && [ "$(jq 'length' "$users")" -gt 0 ]; then
    for user in $(jq -r 'keys[]' "$users"); do
        user_config=$(jq -r ".\"$user\"" "$users")
        storage_path=$(echo "$user_config" | jq -r '.storage_path')

        # Store files for this user
        for file in "$out"/*; do
            if [ -f "$file" ]; then
                store_file "$file" "$user" "$user_config"
            fi
        done
    done
fi
----

== Backend Configuration (backend.toml)

The `backend.toml` file defines your backend's scripts and capabilities.

=== Complete TOML Structure

[source,toml]
----
[mybackend]
# NixOS-specific scripts (at least one of nixos_* or home_* is required)
nixos_check_serialization = "./scripts/nixos-check.sh"
nixos_serialize = "./scripts/nixos-serialize.sh"

# Home Manager-specific scripts
home_check_serialization = "./scripts/home-check.sh"
home_serialize = "./scripts/home-serialize.sh"

# Shared artifact scripts (required if capabilities.shared = true)
shared_check_serialization = "./scripts/shared-check.sh"
shared_serialize = "./scripts/shared-serialize.sh"

# Common scripts (used for both NixOS and Home Manager)
deserialize = "./scripts/deserialize.sh"
check_configuration = "./scripts/check-config.sh"

[mybackend.capabilities]
# Whether this backend supports shared artifacts
shared = true

# Whether this backend actually serializes (vs being a test/mock)
serializes = true

[mybackend.settings]
# Default settings available in $config JSON
storage_path = "/var/lib/mybackend"
encryption = "aes256-gcm"
----

=== Target-Specific Scripts

You can define different scripts for NixOS and Home Manager targets:

[source,toml]
----
[mybackend]
nixos_check_serialization = "./nixos-check.sh"  # Has owner/group
home_check_serialization = "./home-check.sh"    # No owner/group
----

If you define both, the CLI calls the appropriate one based on the target type. If you only define one type (e.g., only `nixos_*`), that backend only works with NixOS configurations.

=== Capabilities Section

[cols="1,1,3"]
|===
| Capability | Default | Description

| `shared` | `false` | Set to `true` if your backend supports `shared_serialize`. Required for shared artifacts.
| `serializes` | `true` | Set to `false` for test backends that don't actually store anything.
|===

=== Include Directive

Backend configurations can be split across multiple files:

[source,toml]
----
# backend.toml
include = ["./backends/agenix.toml", "./backends/sops.toml"]

[my-custom]
check_serialization = "./check.sh"
serialize = "./serialize.sh"
deserialize = "./deserialize.sh"
----

Paths in `include` are resolved relative to the file containing the directive. Nested includes are supported, and circular includes are detected and rejected.

== Environment Variable Reference

This table summarizes all environment variables available to backend scripts:

|===
| Variable | Available In | Description | Example

| `$out` | serialize, shared_serialize | Directory containing generated files from the generator script | `/tmp/artifacts-out-abc123`
| `$inputs` | check_serialization, deserialize | Directory containing JSON files with file metadata (path, owner, group) | `/tmp/artifacts-inputs-def456`
| `$config` | check_serialization, serialize, deserialize, check_configuration | JSON file with backend settings from `[backend.settings]` and per-target config | `{"storage_path": "/var/lib/secrets"}`
| `$machine` | check_serialization, serialize, deserialize, check_configuration | Target identifier - either NixOS machine name or Home Manager user@host | `server-one` or `alice@workstation`
| `$artifact` | All scripts | The artifact name being processed | `ssh-host-key`
| `$machines` | shared_serialize | JSON file mapping machine names to their `artifacts.config.<backend>.nixos.<machine>` configs | `{"server-one": {"key": "abc"}}`
| `$users` | shared_serialize | JSON file mapping user@host to their `artifacts.config.<backend>.home.<user>` configs | `{"alice@host": {"key": "def"}}`
|===

== Error Handling Best Practices

=== Use Strict Mode

Always start scripts with:

[source,bash]
----
#!/usr/bin/env bash
set -euo pipefail
----

This ensures:

* `-e`: Exit immediately on any command failure
* `-u`: Error on undefined variables
* `-o pipefail`: Pipeline fails if any command fails (not just the last)

=== Meaningful Error Messages

When your script fails, print a clear message to stderr:

[source,bash]
----
if [ ! -f "$config" ]; then
    echo "Error: Config file not found: $config" >&2
    exit 1
fi
----

=== Exit Code Conventions

While the CLI only checks zero vs non-zero, follow these conventions:

|===
| Exit Code | Meaning

| 0 | Success
| 1 | General error
| 2 | Invalid arguments or configuration
| 3 | Missing dependencies
| 127 | Command not found
|===

== Testing Your Backend

=== Using the Test Backend as Reference

The nixos-artifacts repository includes reference backend implementations in `pkgs/artifacts/examples/backends/`:

**test backend**::
  Minimal backend that always needs generation and performs no actual storage

**test-shared backend**::
  Demonstrates shared_serialize implementation with JSON parsing

**test-skip-one backend**::
  Demonstrates check_serialization returning different results

=== Testing Phases

1. **Single Artifact Test:**
   Define a simple artifact with one file, test generation and deployment.

2. **Shared Artifact Test:**
   Define a shared artifact across multiple machines or users.

3. **Mixed Target Test:**
   Test with both NixOS and Home Manager configurations.

=== Debugging Tips

* Add `echo` statements to your scripts - they're captured in the TUI's log view
* Check the generated JSON files in temporary directories
* Use `set -x` for verbose trace output during development
* Test scripts manually with mocked environment variables

== Common Patterns

=== Storing in Version Control (agenix-style)

Store encrypted secrets alongside your NixOS configuration:

[source,bash]
----
# serialize.sh
REPO_ROOT=$(git rev-parse --show-toplevel)
SECRETS_DIR="$REPO_ROOT/secrets"

# Encrypt with age and save
age -r "$(cat "$config" | jq -r '.public_key')" -o "$SECRETS_DIR/$artifact.age" "$out/secret"
----

=== Storing in Cloud Services

Example for AWS Secrets Manager:

[source,bash]
----
# serialize.sh
aws secretsmanager create-secret \
    --name "$artifact" \
    --secret-string "$(cat "$out"/secret)" \
    --region "$(jq -r '.region' "$config")"
----

=== Encrypting at Rest

Example using OpenSSL:

[source,bash]
----
# serialize.sh
ENCRYPTION_KEY=$(jq -r '.encryption_key' "$config")
openssl enc -aes-256-cbc -salt \
    -in "$out/secret" \
    -out "$STORAGE_PATH/$artifact.enc" \
    -pass "pass:$ENCRYPTION_KEY"
----

=== Handling Multiple Files

Artifacts can define multiple files:

[source,nix]
----
artifacts.store.my-cert = {
  files = {
    cert = {
      path = "/etc/ssl/cert.pem";
      owner = "root";
      group = "root";
    };
    key = {
      path = "/etc/ssl/key.pem";
      owner = "root";
      group = "ssl-cert";
    };
  };
  # ...
};
----

Your backend will receive both `cert` and `key` files in `$out`.

== Migration Notes

=== From agenix-rekey

If migrating from agenix-rekey:

1. Your existing `.age` files can be read by an agenix backend
2. The `shared = true` pattern is similar to agenix's host-wide secrets
3. Consider using or adapting the `nixos-artifacts-agenix` backend

=== From sops-nix

If migrating from sops-nix:

1. SOPS files can be integrated with a custom backend
2. The `.sops.yaml` configuration maps to `artifacts.config.<backend>.settings`
3. Per-host re-encryption in sops-nix maps to `shared_serialize`

=== From Custom Solutions

If you have an existing custom secrets management solution:

1. Identify your storage mechanism (file system, cloud service, etc.)
2. Write `serialize` and `deserialize` scripts that bridge to your existing system
3. Use `check_serialization` to check your existing storage
4. Gradually migrate artifacts as needed
