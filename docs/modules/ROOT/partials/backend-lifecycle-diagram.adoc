== Backend Execution Lifecycle

The following diagram illustrates the complete lifecycle of backend script execution during artifact generation and deployment.

[mermaid]
----
flowchart TD
    subgraph Generate[Generate Workflow]
        Start([Start Generation]) --> CheckExists{check_serialization}
        CheckExists --"Exit 0 + EXISTS"--> Skip[Skip Generation]
        CheckExists --"Exit non-zero"--> GenerateFiles[Generator produces files]
        GenerateFiles --> SharedCheck{Shared artifact?}
    end

    subgraph Serialize[Serialization Phase]
        SharedCheck --"No"--> SerializeSingle[Run serialize script]
        SharedCheck --"Yes"--> SharedSerialize[Run shared_serialize script]
        SerializeSingle --> StoreSingle[Store for single target]
        SharedSerialize --> StoreMulti[Store for all targets]
    end

    subgraph Deploy[Deployment Phase]
        Activate([System Activation]) --> Deserialize{deserialize script}
        Deserialize --> RestoreFiles[Restore files to $out]
        RestoreFiles --> Install[Install files to target paths]
    end

    StoreSingle --> Complete([Generation Complete])
    StoreMulti --> Complete
    Skip --> Complete
    Install --> Active([Artifact Active])

=== Phase Details

**check_serialization Phase:**

* When: At the start of artifact generation
* Input: Environment variables `$inputs`, `$config`, `$machine`, `$artifact`
* Decision logic:
  ** Exit code 0 with "EXISTS" output → Skip generation (artifact already exists)
  ** Exit code non-zero → Proceed to generation
* Purpose: Prevent overwriting existing secrets without explicit user action

**serialize Phase (for non-shared artifacts):**

* When: After successful generator execution for machine-specific or home-manager artifacts
* Input: Environment variables `$out`, `$config`, `$machine`, `$artifact`
* Purpose: Store the generated files in backend-specific storage
* Called once per target (machine or user)

**shared_serialize Phase (for shared artifacts):**

* When: After successful generator execution for artifacts with `shared = true`
* Input: Environment variables `$out`, `$artifact`, `$machines`, `$users`
* Purpose: Store the artifact once for all targets that share it
* Called once total, receives all target configurations via JSON files

**deserialize Phase:**

* When: During NixOS system activation or home-manager activation
* Input: Environment variables `$inputs`, `$config`, `$machine`, `$artifact`
* Output: Files restored to `$out` directory
* Purpose: Retrieve stored artifacts and make them available at target paths

=== Target Type Handling

Backends must handle two target types:

**NixOS Machines:**
* Machine name available in `$machine`
* Files have `path`, `owner`, and `group` metadata
* Backend config from `artifacts.config.<backend>.nixos.<machine>`

**Home Manager Users:**
* User identifier (format: `user@host`) available in `$machine`
* Files have `path` only (no owner/group)
* Backend config from `artifacts.config.<backend>.home.<user@host>`

The backend.toml configuration supports separate scripts for each target type:

[source,toml]
----
[mybackend]
nixos_check_serialization = "./nixos-check.sh"
nixos_serialize = "./nixos-serialize.sh"
home_check_serialization = "./home-check.sh"
home_serialize = "./home-serialize.sh"
deserialize = "./deserialize.sh"  # Same for both
----
