<objective>
Fix TUI freeze by making event polling non-blocking using spawn_blocking.

Purpose: The async runtime is blocked by synchronous crossterm::event::poll() 
calls, preventing the select! loop from receiving background task results.

Output: Event polling that runs on blocking thread pool, allowing tokio::select!
to concurrently receive channel messages from background tasks.
</objective>

**Root Cause from Debug Session:**
- `poll_next_event()` wraps blocking `events.next_event()` in async block
- `TerminalEventSource::next_event()` calls `crossterm::event::poll()` synchronously
- This blocks the tokio thread for up to 50ms
- `select!` cannot check `res_rx.recv()` while event poll is running
- Serialization completes in background, but result sits in channel
- TUI appears frozen because it can't process the completion

**Fix Required:**
Wrap the blocking event poll in `tokio::task::spawn_blocking()` to run on blocking thread pool.
</context>

    ```rust
        // Wrap blocking crossterm call in spawn_blocking
        // This moves the blocking I/O to a dedicated thread pool
        // allowing tokio::select! to receive channel messages concurrently
        match tokio::task::spawn_blocking(move || {
            events.next_event()
        })
        .await
        {
            Ok(msg) => msg,
            Err(e) => {
                log::error!("Event polling task panicked: {}", e);
                Some(Msg::Tick)
            }
        }
    }
    ```
