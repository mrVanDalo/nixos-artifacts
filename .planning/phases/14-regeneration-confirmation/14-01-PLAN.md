  <action>
    Modify the `CheckSerializationResult` message variant to include an `exists` field.

    Current structure:
    ```rust
    CheckSerializationResult {
        artifact_index: usize,
        result: Result<bool, String>, // Ok(true) = needs generation
        output: Option<CheckOutput>,
    },
    ```

    Change to:
    ```rust
    CheckSerializationResult {
        artifact_index: usize,
        needs_generation: bool,
        exists: bool,  // NEW: true if artifact exists in backend
        result: Result<(), String>, // Err = check failed
        output: Option<CheckOutput>,
    },
    ```

    Also update `SharedCheckSerializationResult` with the same pattern.

    This separates "needs generation" from "exists" - an artifact can:
    - exist=true, needs_generation=false: up to date
    - exist=true, needs_generation=true: stale, needs regeneration (trigger confirmation)
    - exist=false, needs_generation=true: missing, needs generation (no confirmation)

</action>
  <verify>grep -A5 "CheckSerializationResult" pkgs/artifacts/src/app/message.rs | head -20</verify>
  <done>Both CheckSerializationResult and SharedCheckSerializationResult have exists: bool field</done>
</task>

<task type="auto">
  <name>Task 3: Update effect handler to parse exists from check script</name>
  <files>pkgs/artifacts/src/tui/effect_handler.rs</files>
  <action>
    Modify the check_serialization effect handler to determine if artifact exists.

    The check_serialization script returns exit code:
    - 0: artifact is up to date (exists AND current)
    - non-zero: artifact needs generation (may or may not exist)

    Currently the handler only returns "needs generation" boolean. Extend it to:
    1. Capture exit code from check script
    2. Determine `exists` based on script behavior or output parsing
    3. Return both `needs_generation` and `exists` in the result message

    Implementation approach:
    - If exit code is 0: exists=true, needs_generation=false
    - If exit code is non-zero: Check script stderr/stdout for "exists" indicator
      OR assume exists=true for non-zero exit (safer - trigger confirmation)

    Actually, a better approach: Add a convention where check script outputs
    "EXISTS" to stdout when the artifact exists, or use a specific exit code
    (e.g., 2 = exists but stale, 1 = doesn't exist).

    For now, implement the simpler approach:
    - Exit 0: exists=true, needs_generation=false
    - Exit non-zero: Parse output for "EXISTS" keyword
      - If found: exists=true, needs_generation=true
      - If not found: exists=false, needs_generation=true

    Update the `CheckSerializationResult` message construction to include both fields.

</action>
  <verify>
    grep -n "exists" pkgs/artifacts/src/tui/effect_handler.rs | head -20
  </verify>
  <done>Effect handler returns both needs_generation and exists in CheckSerializationResult</done>
</task>
